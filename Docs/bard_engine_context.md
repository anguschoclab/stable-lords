# Bard Narrative Engine: System Context & Audit

The **Bard Engine** is a deterministic, data-driven narrative resolution system designed to decouple simulation outcome logic from literary prose. It allows game systems to request "flavor" for a specific domestic or mechanical event without needing to know the specific strings used.

---

## 1. Design Philosophy
The core tenet of Bard is **"Prose as Prop"**. 
Mechanics (Sim Math) generate **metadata** (Scores, IDs, Tiers). Bard consumes this metadata and returns **context-aware text**.

- **Decoupling**: Writers can update `archive.json` to change the game's "voice" (e.g., making it more gritty vs. more heroic) without changing a single line of TypeScript.
- **Determinism**: By injecting a `RandomGenerator` (PRNG), the engine ensures that a specific combat log entry or news headline remains identical when reloading a save file.
- **Intensity Mapping**: Instead of picking strings at random, Bard maps linear values (0-100) to semantic buckets like `Panned`, `Mixed`, and `Acclaimed`.

---

## 2. Technical Evolution

The system exists in two distinct architectural generations across the codebase:

### Generation 1: Canonical Tokens (Stable Lords)
Used primarily in the Play-by-Play (PBP) combat system.
- **Path**: `src/engine/narrativePBP.ts`
- **Logic**: Uses a specific set of canonical tokens:
    - `%A`: Attacker Name
    - `%D`: Defender Name
    - `%W`: Weapon Name
    - `%BP`: Body Part (resolved via `richHitLocation`)
- **Limitation**: Tightly coupled to "Strike" mechanics (Slashing, Bashing, Piercing).

### Generation 2: Tiered Resolver (Studio Boss)
A generalized, domain-agnostic implementation.
- **Path**: `src/engine/systems/bardResolver.ts`
- **Logic**: Uses a deep hierarchy for lookups:
    - `Domain` (e.g., `Greenlight`)
    - `SubDomain` (e.g., `MarketSat`)
    - `Tone` (e.g., `Trade`, `Tabloid`)
    - `Tier` (e.g., `Risky`, `Prestige`)
- **Innovation**: Uses `{{key}}` syntax and supports a **Dictionary** domain for recursive tag expansion (e.g., generating movie titles).

## 3. Data Schema: `archive.json`

The archive is structured as a nested JSON object.

```json
{
  "Review": {
    "Standard": {
      "Trade": {
        "Acclaimed": ["A tour de force...", "Exquisitely crafted..."],
        "Panned": ["A disastrous miscalculation..."]
      }
    }
  },
  "Dictionary": {
    "ADJECTIVE": ["Silent", "Lethal", "Neon"],
    "NOUN": ["Shadow", "Protocol", "Directive"]
  }
}
```

### Recursive Resolution
When a template contains `{{ADJECTIVE}}`, the engine detects that `ADJECTIVE` is missing from the immediate context and looks it up in the `Dictionary` domain, picking a random variant.

## 4. Design Audit: Approve / Disprove

### APPROVED ✅
1.  **Intensity-to-Tier Mapping**: This is the strongest part of the design. It allows the simulation to provide a raw score (e.g., Review Score: 85) and Bard automatically handles the transition to "Elite" or "Acclaimed" prose.
2.  **Tone Branching**: The ability to request a `Trade` tone for professional feedback and a `Tabloid` tone for social media flavor on the *same event* is highly effective for building world-breadth.
3.  **Deterministic RNG Bindings**: Ensuring the `rng` is passed through every resolution step prevents "Save-Scumming" for better narrative outcomes.

### DISPROVED ❌
1.  **Token Syntax Fragmentation**: Having `%A` in one project and `{{A}}` in another is a major source of friction.
    - *Remedy*: Standardize on double-curly braces `{{~}}` across all implementations.
2.  **Explicit String Dependency**: `domain: 'Greenlight'` and `subDomain: 'Marketing'` are fragile. Typoing a key results in `[MISSING DOMAIN]` at runtime.
    - *Remedy*: Implement an autogenerated `NarrativeKeys` enum or use a build-step validator (Zod) to ensure the code and JSON remain in sync.
3.  **Recursive Depth Hardcoding**: The `limit = 5` in the loop is a "hack" to prevent infinite cycles.
    - *Remedy*: Move to a proper graph-based traversal or at least throw a meaningful error if recursion exceeds depth, rather than silently stopping.
4.  **Loose Context Typing**: The `context: Record<string, any>` allows developers to forget critical keys (like `{{project}}`), resulting in broken strings.
    - *Remedy*: Define `DomainContextMap` interfaces to enforce key presence based on the requested domain.

## 5. Writer's Guide

### Adding New Content
1. **Identify Domain**: Use an existing domain (e.g., `Talent`) or create a new one.
2. **Define Tiers**: Ensure you provide variants for a range of intensities (at least two: a 'Solid' and a 'Risky/Panned' counterpart).
3. **Use Dictionary Tags**: Use existing `{{ADJECTIVE}}` or `{{NOUN}}` tags to add variety without writing infinite templates.
4. **Context Injection**: Remember that keys passed in the code's `context` object (like `actorName`) override Dictionary lookups.

### Example Template Design
> *"Critics are calling {{project}} a {{ADJECTIVE}} masterpiece, cementing {{actor}}'s place in the {{NOUN}} Hall of Fame."*

---

## 6. Performance Considerations
- **Interpolation**: Uses simple Regex. Highly performant for single-line resolutions.
- **Recursive Expansion**: Exponential complexity if Dictionary tags contain other Dictionary tags. 
- **Memory**: The entire `archive.json` is loaded into memory as a static object. For extremely large archives (>5MB), a lazy-loading or SQLite-backed "Bard Content Service" would be required.
